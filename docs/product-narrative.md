# Product Narrative

## Core Idea

ARI is a local-first agentic operating system.

The project is built around the idea that modern AI systems should behave less like isolated chat sessions and more like a durable operational intelligence that maintains continuity across work, projects, routines, and decisions.

ARI is intended to feel like:

- one intelligence
- many quiet processes
- shared memory and state
- explainable reasoning paths
- multiple surfaces over the same underlying system

## Identity

ARI is the public identity and canonical intelligence.

ACE is the interface doctrine and ambient surface layer that shapes how ARI appears across:

- hub UI
- terminal
- Telegram
- notifications
- future desktop companion surfaces
- future voice interfaces

ACE is not a second brain.

## Architectural Direction

The project intentionally prioritizes:

- local-first execution
- explainable state
- deterministic orchestration
- human approval boundaries
- durable memory
- typed shared models
- inspectable execution history

The system is designed so that:

- the home computer hosts the canonical brain
- surfaces consume shared state instead of duplicating logic
- orchestration remains explainable after execution
- future automation can expand gradually without losing operator control

## Career Command

The first proof module is Career Command.

Career Command is an execution workflow focused on:

- job scouting
- role evaluation
- ranking and filtering
- outreach drafting
- approval queues
- execution tracking
- follow-up management

The broader goal is not merely applying to jobs.

The deeper goal is demonstrating that ARI can convert messy, real-world operational work into a structured, inspectable, agent-assisted execution loop.

## Telegram Direction

Telegram is becoming the first practical mobile ACE surface.

The long-term intent is that Telegram can:

- surface updates
- expose approvals
- show execution status
- display pending actions
- route lightweight commands into ARI
- expose state without requiring direct terminal access

This direction allows ARI to become ambient without becoming uncontrolled.

## Inspection Cabinet

A future major direction is the Inspection Cabinet.

The Inspection Cabinet is intended to become the explainability and introspection surface for ARI.

It should expose:

- state
- memory
- orchestration history
- pending approvals
- signals
- alerts
- reasoning paths
- execution logs
- active routines

The core design principle is that ARI should be inspectable rather than magical.

## Companion Surface

A future desktop companion surface is also planned.

The intent is not to create a novelty pet, but to create a lightweight ambient status surface that can visually communicate:

- idle
- working
- blocked
- waiting for approval
- success
- error

without requiring the operator to constantly open the hub.

## Long-Term Direction

ARI is ultimately exploring whether local-first agentic systems can become:

- serious operational tools
- personal execution systems
- automation infrastructure
- career execution engines
- AI implementation systems
- decision-support environments
- business operating layers

The project is intentionally evolving from narrow proof modules toward a broader execution architecture.
